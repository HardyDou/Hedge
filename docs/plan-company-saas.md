## 一、产品概述

### 1.1 产品定位

Hedge Teams 是 Hedge 的企业级团队协作解决方案，提供安全的企业密码管理服务。

**核心价值主张**：
- 零知识架构 — 服务器不存储明文密码
- 简单易用 — 用户越简单越好，打扰越少越好
- 灵活部署 — 支持官方托管 SaaS + 企业自建

### 1.2 目标用户

| 用户群体 | 规模 | 核心需求 |
|---------|------|----------|
| 小团队 | 1-10人 | 密码共享、基本权限 |
| 中团队 | 10-100人 | 分组管理、审计日志 |
| 大团队 | 100人+ | 审批流程、SSO集成 |

### 2.3 零知识架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          零知识数据流                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   用户设备                              服务器                           │
│   ───────                              ──────                           │
│                                                                         │
│   ┌──────────┐                    ┌──────────────┐                     │
│   │ 用户主密码 │◄──────┐      ┌────►│ 加密数据     │                     │
│   └────┬─────┘       │      │     │ (存储)       │                     │
│        │             ▼      │     └──────────────┘                     │
│        ▼        ┌─────────┐  │            │                            │
│   ┌────────┐    │ 派生密钥 │  │            │  服务器从不               │
│   │ 加密密钥 │◄──│ (Argon2)│──┘            │  接触明文                  │
│   └────────┘    └─────────┘                 │                            │
│        │                                    │                            │
│        ▼                                    ▼                            │
│   ┌────────────────────┐            ┌──────────────┐                    │
│   │ 加密/解密 密码     │            │ 加密数据     │                    │
│   │ (本地完成)         │◄──────────►│ 传输/存储    │                    │
│   └────────────────────┘            └──────────────┘                    │
│                                                                         │
│   关键点：                                                             │
│   • 服务器只存储加密数据，无法解密                                      │
│   • 密钥派生在本地完成，密钥不传输                                      │
│   • 成员用自己的主密码解密团队密码                                      │
│   • 成员无需知道团队"主密码"（非托管模式）                            │
└─────────────────────────────────────────────────────────────────────────┘
```

**密钥管理方案**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         密钥管理方案                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   方案：用户主密码派生密钥                                              │
│                                                                         │
│   1. 用户登录：                                                         │
│      用户主密码 → 本地 Argon2id 派生 → 用户加密密钥                    │
│                                                                         │
│   2. 访问团队密码：                                                     │
│      用户加密密钥 + 团队ID → 派生团队密钥 → 解密密码                   │
│                                                                         │
│   3. 多设备同步：                                                       │
│      同一主密码在不同设备派生相同密钥 → 可解密相同数据                  │
│                                                                         │
│   技术选型（技术专家决定）：                                            │
│   • 密钥派生：Argon2id / PBKDF2                                        │
│   • 对称加密：AES-256-GCM                                               │
│   • 密钥长度：256-bit                                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.4 安全邀请机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    零知识邀请方案（完整设计）                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   核心原则：TeamKey 不暴露给服务器                                    │
│                                                                         │
│   加密邀请流程：                                                      │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │  1. 管理员创建邀请：                                        │     │
│   │     ├── 读取目标成员的"身份公钥"（提前注册到服务器）         │     │
│   │     ├── 用公钥加密 TeamKey → EncryptedTeamKey              │     │
│   │     └── 服务器只存储 EncryptedTeamKey（无法解密）           │     │
│   │                                                              │     │
│   │  2. 成员加入：                                              │     │
│   │     ├── 下载 EncryptedTeamKey                               │     │
│   │     ├── 用自己的"身份私钥"解密                              │     │
│   │     └── 获取 TeamKey                                        │     │
│   │                                                              │     │
│   │  ⚠️ 服务器永远不知道 TeamKey                                │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.7 身份密钥注册

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    身份密钥注册（技术前提）                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   为了实现零知识加密，成员需要先注册"身份密钥"：                       │
│                                                                         │
│   注册流程：                                                          │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │                                                              │     │
│   │  用户注册/首次登录时：                                       │     │
│   │  ─────────────────────                                      │     │
│   │  1. 本地生成 RSA 密钥对                                     │     │
│   │     ├── 公钥：IdentityPublicKey                            │     │
│   │     └── 私钥：IdentityPrivateKey (用主密码加密本地存储)    │     │
│   │     │                                                        │     │
│   │     ▼                                                        │     │
│   │  2. 上传公钥到服务器                                         │     │
│   │     POST /users/me/identity-key                            │     │
│   │     {public_key: "..."}                                   │     │
│   │     │                                                        │     │
│   │     ▼                                                        │     │
│   │  3. 私钥本地安全存储                                         │     │
│   │     • 用主密码加密后存入本地数据库                          │     │
│   │     • 永远不传输到服务器                                    │     │
│   │                                                              │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                         │
│   邀请成员时：                                                        │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │                                                              │     │
│   │  1. 从服务器获取成员的公钥                                   │     │
│   │  2. 用公钥加密 TeamKey                                      │     │
│   │  3. 成员用私钥解密                                          │     │
│   │                                                              │     │
│   │  ⚠️ 服务器只有公钥，无法解密                                │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 四、用户体验设计

### 4.1 设计原则

1. **简单优先** — 用户越简单越好，学习成本低
2. **打扰少** — 非必要不打扰用户，安静服务
3. **渐进式** — 用户需要时再展示功能，不强制
4. **统一视图** — 个人密码和团队密码统一展示

## 八、开发计划

### 8.1 Sprint 规划原则

- **Sprint 周期**: 2 周
- **每个 Sprint 有明确交付目标**
- **依赖关系**: 前置 Sprint 完成后才能启动后续
- **MVP 验证**: 每个 Sprint 结束时可演示可用功能

---

### 8.2 MVP 版本（v2.1.0）

#### Sprint 1：基础设施搭建（2周）

**目标**: 后端服务初始化、数据库设计、基础认证

**交付物**:
- [ ] Go 服务器项目初始化，目录结构确定
- [ ] PostgreSQL 数据库 Schema 设计（users, teams, team_members, vault_items）
- [ ] 用户注册/登录 API（JWT 认证）
- [ ] 团队基础表结构设计
- [ ] 本地开发环境 Docker Compose 配置

**完成标准**: 本地可启动服务，用户可注册登录

---

#### Sprint 2：团队核心功能（2周）

**目标**: 团队 CRUD、成员邀请机制

**交付物**:
- [ ] 团队创建 API
- [ ] 团队信息查询/更新 API
- [ ] 团队解散 API
- [ ] 成员邀请 API（邮箱邀请）
- [ ] 邀请链接生成与验证
- [ ] 成员加入/退出团队 API

**完成标准**: 管理员可创建团队并通过邀请链接添加成员

**依赖**: Sprint 1 完成

---

#### Sprint 3：零知识加密架构（2周）

**目标**: 实现团队端到端加密，验证加密/解密流程

**交付物**:
- [ ] TeamKey 生成与分发机制
- [ ] 成员邀请时的密钥加密（使用被邀请人公钥）
- [ ] 团队密码加密/解密流程
- [ ] 客户端加密 SDK 集成
- [ ] 加密流程单元测试

**完成标准**: 可实现团队密码的端到端加密，支持新成员加入时解密历史密码

**依赖**: Sprint 2 完成

---

#### Sprint 4：客户端基础 UI（2周）

**目标**: 客户端团队功能界面

**交付物**:
- [ ] 团队列表页面（展示用户所在团队）
- [ ] 创建团队页面（团队名称、描述输入）
- [ ] 邀请成员页面（选择邀请方式、输入成员信息）
- [ ] 加入团队流程（输入邀请链接/团队码）
- [ ] 团队设置页面（团队信息编辑、解散团队）

**完成标准**: 用户可在客户端完成团队创建、邀请、加入全流程

**依赖**: Sprint 2 完成

---

#### Sprint 5：团队密码协作（2周）

**目标**: 团队密码的 CRUD 操作

**交付物**:
- [ ] 团队密码列表页面（按团队筛选）
- [ ] 密码查看页面（解密显示明文）
- [ ] 密码复制功能（自动复制到剪贴板）
- [ ] 密码添加/编辑页面
- [ ] 密码删除功能
- [ ] 基础权限控制（管理员/成员/只读）

**完成标准**: 团队成员可查看、添加、编辑、删除团队密码

**依赖**: Sprint 3 + Sprint 4 完成

---

#### Sprint 6：数据同步机制（2周）

**目标**: 实现多设备同步，优化离线体验

**交付物**:
- [ ] WebSocket 连接管理（实时推送）
- [ ] 定期拉取同步（1分钟间隔）
- [ ] 冲突检测机制（用户确认模式）
- [ ] 离线操作队列（SQLite + WAL 持久化）
- [ ] 同步状态指示器 UI

**完成标准**: 多设备间数据可同步，冲突时用户可选择保留版本

**依赖**: Sprint 5 完成

---

#### Sprint 7：审计日志与发布准备（2周）

**目标**: 审计功能、测试、Bug 修复

**交付物**:
- [ ] 审计日志 API（登录、密码访问、成员变动等）
- [ ] 审计日志页面（管理员可查看）
- [ ] 集成测试（API + 客户端）
- [ ] 性能优化（WebSocket 连接数、数据库查询）
- [ ] Bug 修复与打磨
- [ ] MVP 版本发布

**完成标准**: v2.1.0 正式发布

**依赖**: Sprint 6 完成

---

### 8.3 专业版（v2.2.0）

#### Sprint 8-9：分组与标签（3周）

**目标**: 密码分组管理、标签功能

**交付物**:

- **竞争分析**: ⚠️ **机遇与挑战并存**。国际巨头（1Password/$LastPass）品牌认知强、功能完善，但定价较高（1Password约$7/人/月）。国内竞品（坚果云/腾讯云密码管理）多缺乏真正的零知识架构。Hedge 的优势：差异化定位（本地优先体验）、私有化部署支持、本土化（钉钉/飞书集成）、CLI 工具。劣势：作为新进入者，企业级功能（SSO/SCIM）待完善，品牌信任度需建立。

- **风险点**: 1）**企业获客周期长**：B2B 销售周期 3-6 个月，需建立销售团队；2）**信任壁垒**：企业采购密码管理极度看重品牌案例和安全性背书；3）**功能深度**：SSO/SCIM/审批流等功能是大型企业刚需，需加速完善；4）**竞争加剧**：大厂可能快速跟进零知识架构。

- **建议**: 1）**聚焦中小团队差异化**：以"安全+简单+低价"切入，避免直接与国际大厂竞争；2）**优先完善企业版核心功能**：审批流、SSO、审计报表是大型企业买单前提；3）**建立安全认证**：获取 SOC2/ISO27001 认证，提升企业信任；4）**开发者社区运营**：通过 CLI 工具吸引开发者口碑传播，降低获客成本；5）**自建版打包销售**：面向大企业提供"软件+服务"打包方案。
